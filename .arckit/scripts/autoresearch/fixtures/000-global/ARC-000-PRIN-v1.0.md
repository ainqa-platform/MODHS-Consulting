## Principles

### P-001: Cloud-First

**Category:** Technology

**Rationale:** Managed cloud services reduce operational overhead, improve elasticity, and accelerate delivery compared to on-premise infrastructure. HM Government's cloud-first policy mandates evaluation of cloud options before considering alternatives.

**Implications:**

- All new services must be designed for cloud deployment (AWS, Azure, or GCP via G-Cloud).
- On-premise hosting requires a formal exception approved by the Lead Architect.
- Infrastructure must be provisioned as code (Terraform or equivalent).

---

### P-002: API-First Design

**Category:** Application

**Rationale:** Exposing all capabilities via well-defined APIs enables reuse, supports future channel expansion (mobile, voice, third-party integration), and decouples front-end from back-end evolution.

**Implications:**

- No capability may be delivered solely through a user interface; a corresponding API must exist.
- APIs must be documented to OpenAPI 3.x standard before go-live.
- Internal APIs are subject to the same standards as public-facing APIs.

---

### P-003: Security by Design

**Category:** Technology

**Rationale:** Security controls embedded from inception are cheaper and more effective than those retrofitted post-build. NCSC Cyber Assessment Framework (CAF) alignment is required for public-facing government services.

**Implications:**

- Threat modelling (STRIDE or equivalent) must be completed at architecture design stage.
- Zero-trust network model: no implicit trust based on network location.
- All data in transit must use TLS 1.2 or above; data at rest must be encrypted.
- Penetration testing required before go-live and annually thereafter.

---

### P-004: Data Minimisation

**Category:** Data

**Rationale:** Collecting only the data necessary for the stated purpose reduces GDPR compliance risk, limits the impact of a data breach, and builds citizen trust.

**Implications:**

- Data requirements must justify each data element with a lawful basis before collection.
- Personal data retention periods must be defined and enforced by automated deletion.
- Data sharing with third parties requires a Data Sharing Agreement and DPIA.

---

### P-005: Open Standards

**Category:** Technology

**Rationale:** Open formats and protocols prevent vendor lock-in, promote interoperability with other government systems, and align with the Government's Open Standards Principles.

**Implications:**

- Prefer open standards (JSON, REST, OAuth 2.0, OpenID Connect) over proprietary equivalents.
- Proprietary formats require an approved exception with documented exit strategy.
- All APIs must support standard authentication protocols (OAuth 2.0 / OIDC).

---

### P-006: User-Centred Design

**Category:** Business

**Rationale:** Services designed around real user needs — evidenced by research — achieve higher adoption, reduce support costs, and meet GDS Service Standard assessment requirements.

**Implications:**

- User research must be conducted before, during, and after service delivery.
- All public-facing touchpoints must meet WCAG 2.2 AA accessibility standard.
- GOV.UK Design System components must be used for citizen-facing interfaces.
- Services must pass a GDS service assessment at Alpha and Beta stages.
